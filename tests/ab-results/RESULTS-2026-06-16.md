## Honest findings (the nuance the eval surfaced — not a clean sweep)

1. **Capable OFF agents catch *statically-derivable* "runtime" defects.** fixture-02 (`.hidden` vs `.hide` class mismatch) and heldout-02 (canvas `chartHeight` omits `MARGIN_BOTTOM`) were both caught by OFF — a sonnet agent reading the source *derived* the symptom without rendering. So the ON advantage is **real but narrower than naive**: the durable ON-only edge is on defects that genuinely cannot be derived from code — **render geometry** (rasterization-dependent: SVG viewBox/symbol windows, canvas overflow) and **execution output** (CLI stdout/stderr). Pick those when seeding render-leak fixtures; avoid statically-obvious ones.

2. **Observation ≠ verdict (KEY).** fixture-03-rebuilt: ON **rendered and precisely observed the clip** ("row 3 bottom 24px cut, row 4 entirely hidden, screenshot confirmed") — and still returned **PASS**, because the neutral intent it was given omitted the "all 12 icons must be fully visible" success criterion, so it rationalized the clip as `overflow:hidden` working "as designed." **Observation surfaces the symptom; the verdict still needs an explicit success criterion.** This independently validates **P0-5 (DoD/Success-Criteria injection)** and argues that `observation-grounding.md` should state: *observation must be paired with an explicit success criterion, or a perfect observer still passes.*

3. **Subtle CSS-overflow defects are hard to seed reliably.** Three fixtures were EXCLUDED (blank primary): fixture-03 r1 (overflow didn't manifest), fixture-03-rebuilt (intent under-specified, finding #2), heldout-03 (a `scale(1.05)` few-px clip too subtle to produce a clear symptom). Render-leak fixtures need a defect with an unambiguous, salient visual symptom.

4. **M4 (untestable): resolved to KEEP with a strict no-boot OFF.** ON marked all 3 unbootable apps **UNTESTABLE** (with a captured blocker, never PASS) and did **not** false-untestable the reachable null — exactly the designed degrade-path. A *first* run came out INCONCLUSIVE for a measurement reason: the OFF rules permitted a *boot attempt*, so capable OFF agents booted, saw the crash, and marked **FAIL** instead of the design-expected **false-PASS** (a measurement-design error — an OFF that can run is not the pre-observation OFF). Re-run with a **strict static-only OFF** (no boot — the faithful pre-4.3.0 behavior): OFF **false-PASSes all 3 unbootable apps** ("a missing DB credential / native build failure is a deployment issue, not a code defect") while ON caps them at UNTESTABLE → margin **+6** → **KEEP**. Lesson: the OFF baseline definition is load-bearing.

## Limitations

- **Mechanism-level, not full pipeline.** Verify-chain isolated under ON/OFF — cleaner signal, but not the literal §7 "run `/harness` 30–40×" protocol.
- **Small n** (4 valid in-author + 3 valid held-out defective fixtures). Margins clear the pre-registered threshold but the sample is small.
- **Model/effort-specific** (sonnet / xhigh). A weaker model might not derive the static defects OFF caught here, *widening* the apparent ON margin; results do not port.

## What this justifies

- **Does:** a measured, pre-registered, positive A/B signal for observation-grounding on render-geometry and execution-output defects, **on both an in-author and an independent-author held-out split**, with 0 false-positives — the design's precondition for considering a CHANGELOG flip (held-out reviewed) is now met for M1.
- **Caveats that must travel with any claim:** mechanism-level (not full-pipeline); the edge concentrates on render-geometry/execution-output (capable OFF derives statically-obvious bugs); observation needs paired success criteria (finding #2); M4 KEEP holds only under a strict no-boot OFF (a loose OFF that boots erases the signal).
